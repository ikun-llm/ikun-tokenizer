# 第二章：词表大小的学问

> 为什么 vocab_size=6400？大了小了会怎样？

![词表大小图示](images/02_vocab.png)

---

## 一句话版本

词表大小（vocab_size）就是分词器的"字典"有多少个词条。太小了表达力不够，太大了模型参数爆炸。6400 是 ikun-2.5B 这种小模型的甜蜜点——够用且不浪费。

---

## 什么是 vocab_size？

```
vocab_size = 分词器能认识多少个不同的 token

就像一本字典：
  小字典（vocab_size=500）：只收录了常用字
    优点：字典薄，好携带
    缺点：遇到生僻词查不到

  大字典（vocab_size=100000）：啥词都收录了
    优点：什么词都认识
    缺点：字典太厚了，背不动

  合适的字典（vocab_size=6400）：常用词都有，太冷门的用基础部件拼
    这就是 ikun 分词器的选择！
```

---

## 太小了会怎样？

```
假设 vocab_size = 256（只认识单个字节）

分词 "鸡你太美"：
  "鸡" → [0xE9, 0xB8, 0xA1]        3 个 token
  "你" → [0xE4, 0xBD, 0xA0]        3 个 token
  "太" → [0xE5, 0xA4, 0xAA]        3 个 token
  "美" → [0xE7, 0xBE, 0x8E]        3 个 token
  ─────────────────────────────────
  总共：12 个 token！4 个字要用 12 个 token 表示

问题：
  1. 序列太长 → 模型处理慢，显存占用多
  2. 模型很难从字节级别理解含义
     （你看到 0xE9, 0xB8, 0xA1 能想到"鸡"吗？模型也很难）
  3. 注意力计算量和序列长度的平方成正比
     序列长 3 倍 → 计算量增加 9 倍！

比喻：
  就像把一首歌拆成一个音一个音来教
  "鸡~你~太~美~" 每个音都单独教
  学生学得慢，还容易学岔了
```

## 太大了会怎样？

```
假设 vocab_size = 100000

分词 "鸡你太美"：
  → [鸡你太美]                     1 个 token！效率很高

但问题来了：

  Embedding 层的参数量 = vocab_size × hidden_dim
  
  vocab_size=6400,  hidden_dim=512 → 6400 × 512 = 327 万参数
  vocab_size=100000, hidden_dim=512 → 100000 × 512 = 5120 万参数

  ikun-2.5B 总共才 2600 万参数！
  光一个 Embedding 层就要 5120 万？比整个模型还大！
  这就是"头太大身子撑不住"

  而且：
    词表中大量 token 在训练数据中出现次数很少
    这些 token 的 embedding 没有被充分训练
    等于白白浪费了参数空间

比喻：
  就像一个练习生背了 10 万个舞蹈动作
  但其中 9 万个一辈子都用不上
  脑子被塞满了，反而记不住重要的动作
```

---

## vocab_size=6400 的设计

```
ikun-2.5B 选择 6400 的理由：

  模型总参数：~2600 万
  Embedding 参数：6400 × 512 = 327 万（占比 12.6%）
  
  这是一个合理的比例。
  对比一下主流模型：

  ┌──────────────┬────────────┬──────────┬──────────────┐
  │ 模型         │ vocab_size │ 总参数    │ Embedding 占比│
  ├──────────────┼────────────┼──────────┼──────────────┤
  │ GPT-2        │ 50257      │ 1.5B     │ ~3.2%        │
  │ LLaMA-7B     │ 32000      │ 7B       │ ~1.8%        │
  │ Qwen-2.5     │ 151936     │ 7B+      │ ~8.8%        │
  │ ikun-2.5B    │ 6400       │ 26M      │ ~12.6%       │
  └──────────────┴────────────┴──────────┴──────────────┘

  大模型可以"奢侈"地用大词表（因为参数总量大）
  小模型必须"精打细算"
  
  6400 能覆盖：
    ✅ 所有常用中文字（约 3000 个）
    ✅ 常用中文词组（鸡你、太美、练习等）
    ✅ 英文常用词和子词
    ✅ 数字、标点、特殊符号
    ❌ 不会收录很罕见的词（用子词拼就行）
```

## 不同 vocab_size 的分词效果对比

```
句子："练习时长两年半的偶像练习生蔡徐坤"

vocab_size=256（字节级）：
  → [e7][bb][83][e4][b9][a0][e6][97][b6][e9][95][bf]...
  → 42 个 token 😱

vocab_size=6400（ikun）：
  → [练习][时长][两年][半的][偶像][练习][生][蔡][徐][坤]
  → 10 个 token ✅

vocab_size=50000（GPT-2 级别）：
  → [练习时长][两年半][的][偶像练习生][蔡徐坤]
  → 5 个 token

vocab_size=150000（Qwen 级别）：
  → [练习时长两年半][的][偶像练习生][蔡徐坤]
  → 4 个 token

观察：
  从 256 到 6400：token 数减少 4 倍！效果显著
  从 6400 到 50000：减少 2 倍，但参数量增加 8 倍
  从 50000 到 150000：只减少 1 个 token，参数量再增 3 倍

  对于小模型来说，6400 就是性价比最高的选择！
```

---

## 如何选择 vocab_size？

```
经验法则：

  模型参数量 < 100M   → vocab_size = 4000~8000
  模型参数量 100M~1B  → vocab_size = 16000~32000
  模型参数量 1B~10B   → vocab_size = 32000~64000
  模型参数量 > 10B    → vocab_size = 64000~150000+

  核心原则：
  Embedding 层占总参数的比例应该在 5%~15% 之间
  太低 → 词表太小，分词太碎
  太高 → 头重脚轻，Embedding 占了太多资源

  ikun-2.5B：26M 参数，6400 词表，占比 12.6% → 合理！
```

## 本章小结

| 概念 | 说明 | 比喻 |
|------|------|------|
| vocab_size | 词表/字典的大小 | 练习生的招式库 |
| 太小 | 序列长，理解难 | 只会单个动作，拼不成舞蹈 |
| 太大 | 参数浪费，训练不充分 | 背了 10 万招但 9 万用不上 |
| 6400 | 小模型的最优平衡点 | 精选招式，够用且精通 |
| Embedding 占比 | 衡量词表是否合适的指标 | 头和身子的比例要协调 |

```
vocab_size 的选择哲学：

  不是越大越好，也不是越小越好
  而是要和模型大小匹配

  小而美，精准高效
  这不就是 ikun 的精神吗？🐔
```
